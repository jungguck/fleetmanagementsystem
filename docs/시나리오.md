# FMS 운영 시나리오 (TurtleBot AGV Fleet)

> 마스터, 이 문서는 `README.md`(개념·아키텍처·로드맵 P0~P5)와 정합하게 **운영 시나리오 8종**을 정리한 것입니다.
> 각 시나리오는 **상황 → 트리거 → 시퀀스(단계별) → 관제/로봇 동작 → 기대결과 → 예외/실패처리** 구조입니다.
> "지게차(적재/하역)" 개념은 README §1 매핑에 따라 **작업 상태머신의 상태(load/unload)** 로 녹였습니다 (포크 없음 → 상태 플래그 / AR마커 / 간이 메커니즘).

---

## 0. 공통 전제 (모든 시나리오 공통)

- **좌표계**: 전 로봇이 동일한 공유 맵(`map` 프레임)을 공유 (README §7-2 로컬라이제이션 일관성).
- **네임스페이스**: 로봇별 `/tb1 /tb2 …` (README §2 통신계층).
- **스테이션 유형**: `PICKUP`(픽업) · `DROP`(드롭) · `CHARGE`(충전) · `IDLE`(대기). Fleet Manager가 맵 위 좌표로 보유.
- **작업 상태머신(Task FSM)** — 지게차 적재/하역을 상태로 표현:

```
 CREATED → ASSIGNED → GOTO_PICKUP → LOADING → GOTO_DROP → UNLOADING → RETURN → DONE
                          │            │          │            │          │
                          └── (주행)   └─(적재상태)└── (주행)   └─(하역상태)└─ 충전/대기 복귀
 취소/실패 경로: * → CANCELLED | FAILED | REASSIGN_PENDING
```

- **로봇 상태(State)**: `IDLE · MOVING · LOADING · UNLOADING · CHARGING · PAUSED · ERROR · ESTOP`.
- **적재 검증(포크 없음 대응, README §7-5)**: 다음 중 택1을 "확인 필요"로 두고 시나리오는 상태 플래그 기준으로 기술 —
  ① 단순 상태 플래그(시뮬 기본) ② AR마커 정렬 확인 ③ 간이 적재 메커니즘/스위치. **채택안은 P5 착수 시 결정(확인 필요)**.

### 상태·색 요약 (GUI_설계.md와 공유)

| State | 색 | 의미 |
|---|---|---|
| MOVING | 🟦 파랑 | nav2 주행 중 |
| IDLE | ⬜ 회색 | 대기 |
| LOADING/UNLOADING | 🟨 노랑 | 적재/하역 |
| CHARGING | 🟩 초록 | 충전 중 |
| ERROR/ESTOP | 🟥 빨강 | 고장/비상정지 |
| PAUSED | 🟧 주황 | 일시정지(장애물 등) |

## 시나리오 1. 단일 배송 (A → B)

| 항목 | 내용 |
|---|---|
| **상황** | 픽업 스테이션 A의 자재를 드롭 스테이션 B로 옮겨야 함. 로봇은 tb1 1대만 가동. |
| **트리거** | 관제 GUI에서 작업 생성: `PICKUP=A, DROP=B` 폼 제출. |

**시퀀스**
```
1. GUI 작업생성 폼 제출 → Fleet Manager 작업큐에 Task#101 (CREATED)
2. 배차: 가용 로봇 tb1 선택 → ASSIGNED (근접/한가 로직, README §3)
3. tb1 ← nav2 goal(A)  ················ State=MOVING, Task=GOTO_PICKUP
4. A 도착 → 적재 상태 진입 ············ State=LOADING, Task=LOADING (적재 플래그 set)
5. tb1 ← nav2 goal(B)  ················ State=MOVING, Task=GOTO_DROP
6. B 도착 → 하역 상태 진입 ············ State=UNLOADING, Task=UNLOADING (적재 플래그 clear)
7. tb1 ← nav2 goal(IDLE 또는 CHARGE) ·· Task=RETURN → DONE
```

- **관제 동작**: 맵뷰에 tb1 라이브 pose·계획 경로 폴리라인 표시, 상태카드 State/Task/배터리 갱신, 작업패널에 Task#101 진행률.
- **로봇 동작**: nav2 `NavigateToPose` 액션으로 A·B 순차 이동, 도착 판정 후 적재/하역 상태 리포트.
- **기대결과**: tb1이 A→B 왕복 후 DONE, GUI 클릭→도착 e2e 성립(README §9 P1 검증과 동일).

**예외/실패처리**
- nav2 경로계획 실패(도달불가): Task=FAILED, 알람 "경로없음", GUI 재시도/취소 버튼.
- A 도착했으나 적재 검증 실패(AR마커 미검출 등): 재정렬 1~2회 재시도 → 실패 시 FAILED + 수동 개입 요청.
- 목표 타임아웃(예: 120s, 확인 필요): PAUSED 후 재계획, 초과 시 FAILED.

## 시나리오 2. 다중 동시작업 + 배차

| 항목 | 내용 |
|---|---|
| **상황** | tb1·tb2·tb3 가동. 서로 다른 3개 배송 작업이 동시 유입. |
| **트리거** | GUI에서 Task#201(A→B), #202(C→D), #203(A→E) 연속 생성 또는 배치 임포트. |

**시퀀스**
```
1. 작업큐: [#201, #202, #203] (CREATED)
2. 배차 루프(주기적): 각 CREATED 작업에 대해
     score = w1·거리(로봇→PICKUP) + w2·(1-배터리) + w3·혼잡도   (가중치 확인 필요)
   최저 score 로봇 배정, 이미 바쁜 로봇 제외
3. #201→tb1, #202→tb2, #203→tb3 배정(예시) → 각자 GOTO_PICKUP
4. 3대 동시 nav2 주행, Traffic Manager가 경로 중첩 구간 감시(시나리오3 참조)
5. 각 로봇 독립적으로 LOADING→GOTO_DROP→UNLOADING→DONE
```

- **관제 동작**: 상태카드 3장 동시 갱신, 맵뷰에 3 pose + 3 경로(로봇별 색), 작업패널에 큐/진행/완료 3열.
- **로봇 동작**: 네임스페이스별 독립 nav2 스택 병렬 구동.
- **기대결과**: 2~3대 동시 주행·작업 완료(README §P2/P3 완료기준: "작업 넣으면 로봇 자동 선택·수행").

**예외/실패처리**
- 가용 로봇 < 작업 수: 초과 작업은 큐에서 대기(CREATED 유지), 로봇 DONE 시 자동 승계.
- 동일 PICKUP(A)에 두 작업 배정: PICKUP 점유 락(1대씩) → 후순위는 GOTO 중 대기(시나리오3의 예약과 연동).
- 배차 tie(동점): 로봇 ID 오름차순 등 결정론적 tiebreak(확인 필요).

## 시나리오 3. 교차로 / 좁은 통로 충돌회피 (최난제, README §7-1)

| 항목 | 내용 |
|---|---|
| **상황** | tb1(동→서), tb2(북→남)가 같은 교차로 셀로 동시 접근. 또는 1폭 통로에서 대향. |
| **트리거** | Traffic Manager가 두 로봇의 계획 경로가 공유 구역(critical zone)에서 겹침을 탐지. |

**시퀀스 (구역 예약 방식 — Open-RMF 대안, README §5/§8 결정포인트)**
```
1. 맵을 critical zone(교차로·좁은통로)로 분할, 각 zone은 mutex(1대 점유)
2. 로봇이 zone 진입 전 Fleet Manager에 예약 요청
3. 예약 승인 순서 = 우선순위(작업긴급도 > 먼저요청 > ID). 미승인 로봇은 zone 앞에서 정지/감속
4. 점유 로봇이 zone 통과·이탈 → 예약 해제 → 대기 로봇 진입 승인
5. 데드락 감지: 상호 대기(A는 B존 기다리고 B는 A존 기다림) → 우선순위 낮은 쪽 후퇴/재계획
```

- **관제 동작**: 맵뷰에 zone 점유 하이라이트, "대기 중(예약)" 상태 표시, 데드락 시 알람.
- **로봇 동작**: nav2 지역 회피(costmap)로 근접 회피 + 상위 예약으로 구역 진입 게이팅. PAUSED 상태로 대기.
- **기대결과**: 교차로에서 안 부딪힘(README §P4 완료기준).

**예외/실패처리**
- **대안 채택**: 커스텀 예약이 불안정하면 **Open-RMF traffic 채택**(README §5, 최난제 기채택 자산). 결정 확인 필요.
- 예약 후 로봇 멈춤(고장): 예약 타임아웃으로 강제 해제 + 해당 로봇 ERROR 처리(시나리오7 연계).
- costmap이 상대 로봇을 정적 장애물로 오인 → 로봇 pose를 동적 obstacle layer로 공유(확인 필요).

## 시나리오 4. 배터리 부족 → 자동 충전복귀

| 항목 | 내용 |
|---|---|
| **상황** | tb2 배터리 25%로 하락(작업 수행 중 또는 대기 중). |
| **트리거** | 로봇 상태리포트의 battery ≤ 임계값(예: LOW=30%, CRIT=15%, 확인 필요) 도달. |

**시퀀스**
```
1. Fleet Manager가 tb2 battery ≤ LOW 감지
2. tb2가 작업 중이면: 현재 작업 완료 후 신규 배차 제외(soft) → CHARGE 스테이션 예약
   tb2가 CRIT면: 현재 작업 중단·재배차(시나리오7) → 즉시 CHARGE 복귀(hard)
3. tb2 ← nav2 goal(가장 가까운 빈 CHARGE 스테이션)  State=MOVING
4. 도착 → State=CHARGING (도킹은 상태 플래그/간이 도킹, 확인 필요)
5. battery ≥ 복귀 임계(예: 80%, 확인 필요) → State=IDLE, 배차 대상 복귀
```

- **관제 동작**: 카드 배터리 바 노랑/빨강, "충전복귀" 배지, 맵뷰에 CHARGE로 향하는 경로.
- **로봇 동작**: 진행 작업 정리 후 충전소 이동, 충전 중 배차 제외.
- **기대결과**: 방전 없이 자동 충전복귀·복귀(README §P5, "왕복 사이클 무인 반복").

**예외/실패처리**
- 모든 CHARGE 점유: 대기열 등록, 가장 배터리 높은 충전 로봇을 조기 방출(확인 필요) 또는 IDLE에서 대기.
- CRIT인데 충전소 도달 전 방전 예측: 최근접 안전지대 정지 + 알람 "방전임박, 수동회수".
- 도킹 실패(정렬): 재도킹 재시도 → 실패 시 ERROR + 수동.

## 시나리오 5. 이동 중 장애물 / 사람 감지 (안전, README §7-7)

| 항목 | 내용 |
|---|---|
| **상황** | tb1 주행 중 경로 상에 사람/예상치 못한 장애물 진입. |
| **트리거** | LiDAR/costmap이 안전거리 내 동적 장애물 감지. |

**시퀀스**
```
1. nav2 costmap 갱신 → 안전거리(예: 0.5m, 확인 필요) 침범 감지
2. 감속 → 정지: State=PAUSED (속도제한/정지, README 안전)
3. 장애물 지속: 지역경로 재계획(우회) 시도
4. 우회 불가·통로 막힘: 일정시간(예: 10s) 대기 후 재시도, 알람 "장애물 대기"
5. 장애물 소거 → 자동 재개 State=MOVING
```

- **관제 동작**: 카드 PAUSED(주황), 맵뷰에 장애물 위치 마커, 알람 로그.
- **로봇 동작**: nav2 collision monitor/속도제한, 정지 후 자율 재개.
- **기대결과**: 충돌 없이 정지·우회·재개, 사람 안전 보장.

**예외/실패처리**
- 장애물 장시간 지속(예: 60s): Task=REASSIGN_PENDING 또는 관제 개입 요청.
- 센서 노이즈로 오탐 잦음(시뮬↔실기 갭 §7-9): 필터/최소지속시간(debounce) 적용(확인 필요).
- 사람 근접 시 추가 안전: 최고속도 하향·경고음(실기, 확인 필요).

## 시나리오 6. 작업 우선순위 변경 / 재배차

| 항목 | 내용 |
|---|---|
| **상황** | 대기 중인 저순위 작업들 사이에 긴급 작업이 삽입되거나, 진행 작업의 우선순위가 상향. |
| **트리거** | GUI 작업패널에서 우선순위 변경 또는 긴급 작업 생성(priority=HIGH). |

**시퀀스**
```
1. Task#301(HIGH) 생성 → 작업큐 우선순위 재정렬(HIGH 선두)
2. 배차 재평가:
   - 가용 IDLE 로봇 있으면 즉시 배정
   - 없으면: 진행 중 저순위 작업 선점 여부 판단(preempt 정책, 확인 필요)
3. 선점 시: 대상 로봇의 현재 작업 → REASSIGN_PENDING(큐 복귀), 로봇은 #301로 전환
4. 선점된 작업은 다른 가용 로봇/차기 슬롯에 재배차
```

- **관제 동작**: 작업패널 큐 순서 실시간 변화, "재배차" 배지, 영향받은 카드 강조.
- **로봇 동작**: 현재 goal 취소(nav2 cancel) → 신규 goal 수령.
- **기대결과**: 긴급 작업이 우선 처리되고 나머지는 손실 없이 재배차.

**예외/실패처리**
- 선점 금지 작업(적재 중 LOADING): 원자적 단계 완료 후 전환(적재물 방치 방지).
- 재배차 진동(같은 작업 반복 이동): hysteresis/최소유지시간으로 억제(확인 필요).
- 우선순위 동률 폭주: 큐 상한·거절 정책(확인 필요).

## 시나리오 7. 로봇 고장 시 작업 재할당

| 항목 | 내용 |
|---|---|
| **상황** | tb3가 작업 중 통신두절/노드죽음/모터오류로 응답 중단. |
| **트리거** | 상태리포트 하트비트 타임아웃(예: 3회 연속 미수신, 확인 필요) 또는 로봇 자체 fault 리포트. |

**시퀀스**
```
1. Fleet Manager 하트비트 미수신 → tb3 State=ERROR, 배차 대상 제외
2. tb3의 진행 작업 판단:
   - 적재 전(GOTO_PICKUP): Task→REASSIGN_PENDING → 큐 복귀 → 타 로봇 재배정
   - 적재 후(GOTO_DROP, 적재물 보유): 물리 회수 필요 → 관제 개입 알람 + 수동 회수 태스크
3. tb3 점유 예약/구역 강제 해제(시나리오3 연계) → 교통 정상화
4. 복구 시 tb3 재접속 → 재로컬라이즈 확인 후 IDLE 복귀
```

- **관제 동작**: 카드 ERROR(빨강)·연결 "끊김"(재사용: components.robot_card의 끊김 테두리/배경 패턴), 알람 "로봇 고장", 재배차 로그.
- **로봇 동작**: (가능하면) 안전정지, 상태 상실.
- **기대결과**: 단일 로봇 고장이 fleet 전체를 멈추지 않고 작업이 이어짐.

**예외/실패처리**
- 적재물 보유 로봇 고장: 자동 재배차 불가 → 수동 회수 워크플로(확인 필요).
- 오탐(일시 네트워크 지연): 하트비트 grace + 재시도(robot_api RobotManager 재접속 패턴 참조) 후에만 ERROR 확정.
- 재접속 후 pose 드리프트: 재로컬(AMCL 초기포즈) 요구(README §7-2).

## 시나리오 8. E-STOP (전체 정지, README §7-7)

| 항목 | 내용 |
|---|---|
| **상황** | 안전사고/긴급상황으로 fleet 전체 즉시 정지 필요. |
| **트리거** | GUI 상단 **E-STOP 버튼** 클릭(또는 물리 버튼/안전 이벤트). |

**시퀀스**
```
1. GUI E-STOP → Fleet Manager broadcast(전 네임스페이스)
2. 각 로봇 즉시 정지: cmd_vel=0, nav2 작업 취소, State=ESTOP
   (병렬 전파 — robot_api `/api/fleet/emergency_stop` + RobotManager.units 병렬 패턴 재사용)
3. 부분 실패 허용: 응답 못한 로봇은 개별 알람(partial-failure 집계)
4. 모든 작업 HOLD(취소 아님) — 해제 대기
5. 해제(RESET): 관제가 명시적 RESET → 안전확인 → 각 로봇 IDLE 복귀 → 작업 재개 여부 선택
```

- **관제 동작**: 화면 전체 빨강 배너, 전 카드 ESTOP, 해제 버튼 비활성→확인 후 활성.
- **로봇 동작**: 최우선 정지(다른 명령보다 우선), 해제 전까지 명령 무시.
- **기대결과**: 1초 내 전 로봇 정지(목표, 확인 필요), 부분실패 가시화, 명시적 해제로만 복귀.

**예외/실패처리**
- 일부 로봇 미응답: 병렬 정지 결과 dict로 성공/실패 표기(robot_api start_all/stop_all 반환 패턴), 미응답 로봇 물리 확인 요청.
- 네트워크 분할로 broadcast 누락: 로봇 측 워치독(관제 하트비트 끊기면 자율 정지, 확인 필요)로 이중화.
- 오작동 방지: E-STOP은 즉시, RESET은 2단계 확인(오해제 방지).
