# FMS 운영 시나리오 (turtlesim AGV Fleet)

> `README.md`(개념·구조)와 정합하게 **운영 시나리오 8종**을 정리한 문서.
> 각 시나리오는 **상황 → 트리거 → 시퀀스 → 관제/로봇 동작 → 기대결과 → 예외처리** 구조.
> 각 시나리오 제목에 현재 상태를 표기한다: **[구현됨]** / **[일부]** / **[미구현]**
> — 미구현 시나리오는 "이렇게 만들 것"이라는 설계이지 동작 설명이 아니다(혼동 금지).
>
> 지게차의 "적재/하역"은 **작업 상태머신의 단계**로 녹였다(거북이엔 포크가 없다).

## 0. 공통 전제

- **좌표계**: turtlesim 창의 좌표 그대로 — `0 ~ 11.09` 정사각 평면. 전 로봇이 공유.
  (SLAM/AMCL 없음. `/turtleN/pose` 가 오차 없는 실측을 준다 → 로컬라이제이션 문제 자체가 없다.)
- **네임스페이스**: 로봇별 `/turtle1 /turtle2 …` (= config 의 `id`).
- **스테이션 유형**: `pickup` · `drop` · `charge`. `gui/config.yaml` 이 좌표로 보유.
- **가상벽**: config 의 `walls` = **진입금지 구역**. turtlesim 엔 실제 장애물이 없고,
  A* 플래너만 아는 개념이다(= nav2 costmap 의 역할을 대신).

### 로봇 상태 (`models.STATE_COLORS` 와 1:1 — 이게 전부다)

| State | 색 | 의미 |
|---|---|---|
| `idle` | ⬜ 회색 `#9aa7b0` | 대기 (배차 가능) |
| `driving` | 🟦 파랑 `#2e86de` | A* 경로 자동주행 중 |
| `manual` | 🟩 청록 `#16a085` | 수동 teleop 조작 중 |
| `charging` | 🟩 초록 `#27ae60` | 충전 중 |
| `error` | 🟥 빨강 `#e74c3c` | 에러 (※ 현재 이 상태로 만드는 코드는 없음) |

`online` 은 상태와 별개 플래그: `/turtleN/pose` 가 2초(`POSE_TIMEOUT`) 이상 끊기면 `False`.

### 작업(Task) 상태머신 — `state.py` 구현 그대로

```
 pending ──배차(dispatch)──> running ──픽업도착──> running ──드롭도착──> done
    ▲                        phase=to_pickup      phase=to_drop
    └──── 재배차(requeue) ────┴─ 로봇 통신두절/에러, 또는 도착 전에 멈춤(정지·teleop)
 E-STOP: pending|running → cancelled
```

> ⚠ 실제 구현은 이 4상태(`pending/running/done/cancelled`)뿐이다.
> `LOADING`/`UNLOADING`/`RETURN` 은 **별도 상태가 아니라** 로봇 카드의 작업 설명 문자열
> (`"PICK-A→DROP-B (적재)"`)로만 표현된다. 적재 검증(AR마커 등)은 없다.

> ⚠⚠ **`idle` 은 '도착' 이 아니다.** 정지(■)·teleop 만료·통신두절로도 `idle` 이 된다.
> 그래서 `_advance_tasks()` 는 "로봇이 idle 이다"가 아니라
> **"목표 스테이션 반경 `ARRIVE_EPS`(0.4) 안에 실제로 서 있다"** 를 확인하고서야 다음 단계로 넘긴다.
> 아니면 `_requeue()` 로 작업을 큐에 돌려보낸다(다른 로봇이 이어받음).
> (이 검증이 없으면 주행 중에 ■ 를 두 번 누르는 것만으로 작업이 "완료"된다 — 실제 있었던 버그.)

## 시나리오 1. 단일 배송 (A → B) **[구현됨]**

| 항목 | 내용 |
|---|---|
| **상황** | PICK-A 의 자재를 DROP-B 로 옮겨야 함. |
| **트리거** | 웹 GUI "➕ 작업 생성" 폼에서 `픽업=PICK-A, 드롭=DROP-B` 추가. |

**시퀀스**
```
1. GUI 폼 제출 → FleetState.add_task() → 작업큐에 Task#1 (pending)
2. refresh(0.5s) 의 dispatch() → 가용 로봇 중 PICK-A 최근접 선택
     → running / phase=to_pickup, 로봇 task="PICK-A→DROP-B (픽업)"
3. send_to_station(PICK-A) → planner.plan() A* 경로 → RobotState.path
     → 20Hz 제어루프가 cmd_vel 로 추종        State=driving
4. PICK-A 도착(path 소진) → State=idle
5. _advance_tasks() 가 idle 을 감지 → phase=to_drop, send_to_station(DROP-B)
     로봇 task="PICK-A→DROP-B (적재)"          State=driving
6. DROP-B 도착 → State=idle → _advance_tasks() → Task#1 = done
```

- **관제 동작**: 맵뷰에 라이브 pose·A* 계획경로 점선·펜 궤적, 상태카드 갱신, 작업큐에 진행 표시.
- **로봇 동작**: `/turtleN/cmd_vel` 로 제자리회전(`|err|>0.12rad`) → 전진 추종, 도착 판정 `<0.15`.
- **기대결과**: 거북이가 PICK-A → DROP-B 순차 이동 후 Task done. **실측 확인됨.**

**예외/실패처리**
- A* 경로 없음(도달 불가): `planner.plan()` 이 **빈 리스트**를 반환 → `send_goal()` 이 목표를 **거부**하고
  UI 가 "경로가 없습니다 (벽에 막힘)" 알림. 배차도 하지 않는다(작업은 pending 유지 → 재시도).
- 도착 전에 멈춤(운영자가 ■ 또는 teleop): 작업은 `pending` 으로 되돌아가 **재배차**된다. **구현됨.**
- 로봇 통신두절: `error` 처리 + 작업 재배차(시나리오7). **구현됨.**
- ⚠ **목표 타임아웃은 없다** — 로봇이 계속 `driving` 인 채로 목표에 못 닿으면(예: 창 경계에 끼임)
  작업은 `running` 에 머문다. 시간 제한을 두는 건 미구현.

## 시나리오 2. 다중 동시작업 + 배차 **[구현됨]**

| 항목 | 내용 |
|---|---|
| **상황** | 거북이 4마리 가동. 여러 배송 작업이 동시 유입. |
| **트리거** | GUI 에서 작업 연속 생성. |

**시퀀스**
```
1. 작업큐: [#1, #2, #3] (pending)
2. dispatch() 매 0.5s: 각 pending 작업에 대해
     가용 = state==idle AND online AND battery > LOW_BATT(20%)
     선택 = 픽업 스테이션까지 '유클리드 거리 최소' 로봇   ← 단순 근접
3. 가용 로봇이 없으면 break → 다음 틱에 재시도(작업은 pending 유지)
4. 각 로봇 독립적으로 to_pickup → to_drop → done
```

- **기대결과**: 여러 대 동시 주행·작업 완료. **실측 확인됨**(작업 생성 → 최근접 로봇 자동 배차).

**예외/실패처리**
- 가용 로봇 < 작업 수: 초과 작업은 pending 대기, 로봇이 idle 되면 자동 승계. **구현됨.**
- 동일 PICK-A 에 두 작업: **점유 락 없음** → 두 거북이가 같은 좌표로 간다(겹침). 미구현.
- 배차 tie(거리 동점): `min()` 이 먼저 온 로봇을 고름 → config 순서 기준 결정론적.

## 시나리오 3. 교차로 / 충돌회피 **[구현됨 — 단, 정면대향은 한계]**

| 항목 | 내용 |
|---|---|
| **상황** | 두 거북이가 같은 지점으로 동시 접근. |
| **트리거** | 20Hz 제어루프가 매 틱 `traffic.yielders()` 호출 — 거리 < `safe_dist`(config, 기본 1.2). |

**규칙 (`gui/traffic.py`)**
```
1. 움직이는(driving/waiting) 두 로봇의 거리가 safe_dist 안 → 충돌 위험
2. 우선순위 낮은 쪽(prio 큰 쪽 = config 목록 아래쪽)이 양보
   · prio 는 config 순서로 배정. 동률이면 id 로 tiebreak → 항상 결정론적
3. 단, 양보 대상이 상대 '쪽으로 다가가는 중'일 때만 세운다
   (내적 > 0 검사. 멀어지는 중이면 세울 이유가 없다 — 뒤따라가기가 막히면 안 되니까)
4. 위험이 풀리면 자동 재개 — 경로(path)를 들고 서 있었으므로 가던 길 그대로
```

**왜 교착(deadlock)이 없나 — 이게 설계의 핵심**
우선순위가 **전순서(total order)** 라서 "A는 B를, B는 A를 기다리는" 상황이 **구조적으로 불가능**하다.
두 로봇 중 항상 정확히 한 쪽만 선다. → 교착 감지·해소 코드가 아예 필요 없다.
(구역 예약 방식은 유연하지만 교착 감지를 따로 만들어야 한다. 여기선 교착이 생길 수 없는 규칙을 골랐다.)

- **관제 동작**: 위험 쌍을 맵에 **빨간 점선**으로 연결, 양보 중인 로봇에 **주황 안전거리 원**,
  상태 `waiting`(주황), 상단 요약에 `⚠양보대기 N`.
- **기대결과**: 교차 상황에서 한 대가 서고 다른 대가 지나간 뒤 재개. **실측 확인됨.**

**⚠ 한계 — 정직하게 (실측: 정면대향 시 최소 접근거리 0.15)**
- **정면으로 마주치면 겹쳐 지나간다.** 양보하는 쪽은 서지만, **선 자리가 곧 상대의 경로**라서
  우선순위 로봇이 그대로 통과한다. "충돌을 막는다"가 아니라 **"관제가 교통을 정리한다"** 까지가
  지금 수준. (turtlesim 은 거북이끼리 물리 충돌이 없어 겹쳐도 아무 일도 안 일어난다.)
- 제대로 풀려면: 선 로봇을 **임시 장애물로 넣고 우선순위 로봇이 A\* 재계획**(우회) —
  또는 통로에 일방통행 규칙. 둘 다 미구현.
- **서 있는(idle) 로봇은 장애물로 안 친다.** 치면 그 앞 로봇이 영원히 못 간다(비켜줄 수가 없으므로).

## 시나리오 4. 배터리 부족 → 자동 충전복귀 **[구현됨]**

| 항목 | 내용 |
|---|---|
| **상황** | 거북이 배터리가 20% 아래로 하락. |
| **트리거** | `_auto_charge()` 가 매 0.5s 검사: `state==idle AND battery < LOW_BATT(20%)`. |

> ⚠ **배터리는 가상이다.** turtlesim 엔 배터리 개념이 없어서 FMS 관제(충전복귀)를 보여주려고
> 코드가 만들어 낸 값이다. 주행 중 `0.5%/s` 소모, 대기 중 `0.04%/s` 자연방전, 충전 `4%/s` 회복.

**시퀀스**
```
1. _auto_charge() 가 저전력 idle 로봇 감지
2. send_goal(CHARGE 좌표, kind="charge") → task="저전력→충전복귀"   State=driving
3. 도착(path 소진) → goal_kind=="charge" 이므로 State=charging, task="충전중"
4. battery >= 99.5% → State=idle → 배차 대상 복귀
```

- **관제 동작**: 카드 배터리 바가 빨강(<20%), 맵뷰에 CHARGE 로 향하는 점선 경로.
- **기대결과**: 방전 없이 자동 충전복귀 후 작업 복귀.

**예외/실패처리**
- **작업 중(running) 로봇은 충전 안 감**: 조건이 `state==idle` 이라, 작업 수행 중엔 배터리가
  0%가 돼도 계속 간다(배터리가 가상이라 실제로 멈추진 않는다). 의도된 단순화.
- **E-STOP 중엔 충전복귀도 안 한다**(`estopped` 래치). 이게 없으면 E-STOP 을 눌러도
  저전력 로봇이 0.5초 뒤 충전소로 스스로 출발한다 — E-STOP 이 아니게 된다(실제 있었던 버그).
- CHARGE 스테이션은 **1개·점유 개념 없음** → 여러 대가 같은 충전소로 겹쳐 간다.
- config 에 `kind: charge` 스테이션이 없으면 `_auto_charge()` 는 조용히 아무것도 안 한다.

## 시나리오 5. 장애물 / 사람 감지 **[해당 없음]**

turtlesim 에는 **센서(LiDAR)도 동적 장애물도 사람도 없다.** 이 시나리오는 성립하지 않는다.

대응되는 유일한 개념이 **가상벽**(config `walls`)이고, 이건 "감지해서 피하는" 게 아니라
**A\* 가 처음부터 피해서 경로를 만드는** 방식이다(정적 회피).

- `planner.py` 의 `inflate=1`(0.2 단위) 팽창으로 벽에 붙지 않게 여유를 둔다.
- 실제로 벽을 밟아도 아무 일도 안 일어난다(물리 없음) — 관제가 안 가게 만들 뿐이다.
- ⚠ **turtlesim 창 경계(0/11.09)에 부딪히면 거북이가 멈추고** `Oh no! I hit the wall!` 로그가 뜬다.
  → 스테이션·초기위치를 경계에서 띄워 두는 이유(config 참고).

## 시나리오 6. 작업 우선순위 변경 / 재배차 **[미구현]**

- **현재**: 작업에 `priority` 필드가 **없다**. 큐는 생성순(FIFO)이고, `dispatch()` 는 리스트 앞에서부터 훑는다.
  진행 중 작업의 선점(preempt)도 없다. 취소는 **E-STOP 으로 전부 취소**만 가능(개별 취소 없음).
- **만든다면**: `Task.priority` 추가 → `dispatch()` 에서 pending 을 우선순위 정렬 →
  가용 로봇 없으면 저순위 running 작업 선점(로봇 stop → 작업 pending 복귀) → 고순위 배차.
- **주의**: 재배차 진동(같은 작업이 로봇 사이를 왔다갔다) 방지에 최소유지시간 필요.

## 시나리오 7. 로봇 고장 시 작업 재할당 **[구현됨]**

| 항목 | 내용 |
|---|---|
| **상황** | 거북이가 죽음(`ros2 service call /kill`) 또는 turtlesim 종료 → pose 두절. |
| **트리거** | `_control()` 이 매 틱 검사: `now - last_pose > 2.0s` → `online=False`. |

**시퀀스**
```
1. pose 2초 미수신 → online=False
   (단, '기동 직후 첫 pose 대기중'은 제외 — was_online 이었다가 끊긴 경우만 고장으로 본다)
2. state="error", path=[], task="통신두절"
     → 맵에 유령 계획경로가 계속 그려지는 것 방지
3. 카드에 "⚠ OFFLINE — /pose 없음" + 빨강, 맵뷰에서 거북이 흐려짐(opacity 0.35)
4. dispatch() 는 online+idle 로봇만 고름 → 죽은 로봇에 신규 배차 안 됨
5. _advance_tasks() 가 그 로봇의 running 작업을 pending 으로 되돌림 → 다음 dispatch() 가 재배차
```

- **기대결과**: 한 대가 죽어도 그 작업이 다른 거북이에게 넘어가 완주한다.
- **복구**: `error` 는 자동으로 안 풀린다. 운영자가 **■(정지)** 를 눌러 `idle` 로 되돌린다
  (되살아난 로봇이 옛 명령으로 갑자기 움직이지 않게 하려는 의도적 설계).
- **미구현**: 알람 로그, 자동 재접속 감지.

## 시나리오 8. E-STOP (전체 정지) **[구현됨 — 단순화]**

| 항목 | 내용 |
|---|---|
| **상황** | 긴급 상황으로 fleet 전체 즉시 정지. |
| **트리거** | GUI 조작 패널 우측 **[🟥 E-STOP]** 클릭. |

**시퀀스 (`FleetState.stop_all()`)**
```
1. estopped = True   ← 래치. 이게 핵심.
2. 전 로봇: path=[] , manual=None , goal_kind="" , state=idle , task="-"
3. 다음 제어틱(≤50ms)에 cmd_vel=0 이 발행되어 실제로 멈춤
4. 진행/대기 작업(state != done) 전부 → cancelled
5. 래치가 풀릴 때까지 dispatch()·_auto_charge()·send_goal()·teleop() 전부 거부
6. 운영자가 [🔓 해제(RESET)] → reset() → estopped=False → 조작 재개
```

- **기대결과**: 클릭 즉시 전 거북이 정지 + 작업 취소 + **가만히 있음**. **실측 확인됨.**
- **⚠ 래치가 왜 필요했나 (실제 있었던 버그)**: 래치 없이 `idle` 로만 떨어뜨렸더니,
  0.5초 뒤 `_auto_charge()` 가 저전력 로봇을 충전소로 보내고 `dispatch()` 가 남은 작업을
  배차해 **로봇들이 스스로 다시 출발했다.** 정지시켰는데 움직이면 그건 E-STOP 이 아니다.
- **설계상 단순화 (원안과 다른 점)**:
  - 작업을 **HOLD 가 아니라 cancelled** 로 만든다(재개 불가, 새로 만들어야 함).
  - `ESTOP` 이라는 **로봇 상태는 없다** — `idle` + 전역 `estopped` 플래그로 표현.
  - RESET 2단계 확인 없음(1클릭). 부분실패 집계 없음(발행 실패를 확인하지 않는다).
  - ■(개별 정지)는 E-STOP 중에도 허용 — 정지는 언제나 안전한 명령이라서.
- **왜 단순한가**: 로봇이 전부 같은 프로세스 안의 거북이라 broadcast·네트워크 분할·
  미응답 같은 실 fleet 의 문제가 없다. 실기라면 위 항목들은 반드시 필요하다.
