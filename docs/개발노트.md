# 개발 노트 — 시스템 구조를 어떻게 만들었나

> 이 문서는 **"왜 이렇게 만들었나"** 와 **"어디서 물렸나"** 를 남긴 기록이다.
> `README.md`(무엇인지) · `GUI_설계.md`(화면) · `시나리오.md`(동작) 은 결과를 설명하고,
> 이 문서는 **결정과 실패**를 설명한다. 코드를 읽어선 알 수 없는 것만 적는다.

## 1. 출발점을 갈아엎은 결정: TurtleBot3 → turtlesim

**처음 설계**는 TurtleBot3 + Gazebo + nav2 + SLAM 이었다(`fms_ws/`, `setup/install.sh`).
그런데 원래 원했던 건 **ROS 2 의 기본 예제인 turtlesim** 이었다. 그래서 통째로 갈아엎었다.

**이 결정으로 사라진 것** (= 안 만들어도 되게 된 것):

| 원래 필요했던 것 | turtlesim 에선 |
|---|---|
| SLAM 으로 맵 작성 → 저장 | 필요 없음 (평면 좌표 0~11.09 가 곧 맵) |
| AMCL 로컬라이제이션·드리프트·재로컬 | **`/turtleN/pose` 가 오차 없는 실측을 그냥 준다** |
| nav2 파라미터 튜닝, costmap | A* 직접 구현 (`planner.py`) |
| 네임스페이스별 nav2 스택, tf prefix | 토픽 이름만 `/turtleN/…` |
| Gazebo world, 물리, 센서 노이즈 | 없음 |

**대가**: LiDAR·동적 장애물·물리·실기 갭이 전부 사라졌다.
**얻은 것**: FMS 의 진짜 가치인 **관제 두뇌**(배차·작업 FSM·경로계획·충전·안전·교통)에만 집중.
저 문제들은 "로봇 문제"지 "관제 문제"가 아니다. 관제 로직은 2D 평면 위에서 전부 만들고 검증할 수 있다.

> **판단**: 3D 물리를 흉내내는 것보다, 관제 로직을 **끝까지 정확하게** 만드는 게 낫다.
> 나중에 실 로봇이 필요해지면 `poll()/send_goal()` 인터페이스에 ROS 백엔드를 갈아끼우면 된다.

## 2. 구조를 이렇게 나눈 이유

```
config.yaml   ← 로봇·스테이션·벽·safe_dist  (대수 조정은 코드 수정 없이 여기서만)
    ↓
models.py     ← RobotState·Task·색·FLEET_LOCK   [최하위: 아무것도 import 안 함]
    ↓
planner.py (A*)   traffic.py (교통관제)          [순수 로직: ROS·UI 모름]
    ↓
turtlesim_source.py  ← 로봇의 '몸' (ROS: spawn/pose/cmd_vel 20Hz)
    ↓
state.py      ← 관제 '두뇌' (배차·작업FSM·충전·E-STOP) + sim2d 폴백
    ↓
ui/*          ← 화면 (맵뷰·카드·조작·작업큐)
```

### 왜 `models.py` 를 따로 뺐나 — 순환 import
처음엔 `RobotState` 가 `state.py` 안에 있었다. 그런데:
`state.py` 가 `TurtlesimSource` 를 import → `turtlesim_source.py` 가 `RobotState` 를 import → **순환**.
→ 아무것도 의존하지 않는 최하위 모듈로 모델·색·락을 내렸다. 백엔드와 두뇌가 **같은 모델을 공유하되 서로를 모른다**.

### 왜 백엔드를 인터페이스(`poll`/`shutdown`)로 추상화했나
`config.yaml` 의 `source:` 한 줄로 `turtlesim` ↔ `sim2d` 를 바꾼다.
- **ROS 가 없는 PC 에서도 UI 를 볼 수 있다** (sim2d 폴백).
- 나중에 실 로봇/Gazebo 를 붙일 때 갈아끼울 자리가 이미 있다.
- 대신 **두 백엔드의 동작이 갈라지지 않게** 교통관제 같은 규칙은 `traffic.py` 로 빼서 둘 다 쓴다.

### 왜 `planner.py` / `traffic.py` 는 순수 함수인가
ROS 도 UI 도 모른다 → **혼자 테스트할 수 있다**. 실제로 `traffic.yielders()` 는
거북이 없이 좌표만 넣어 검증했다(e2e3 의 마지막 케이스).

## 3. 가장 중요한 구조 결정: **주기가 2개다**

| 주기 | 누가 | 무엇 | 왜 이 속도 |
|---|---|---|---|
| **20Hz** | ROS 실행기 스레드 (`turtlesim_source._control`) | cmd_vel 발행·도착판정·교통관제·배터리 | 느리면 거북이가 뚝뚝 끊긴다 |
| **0.5s** | `app.timer` → `state.refresh()` | 배차·작업FSM·충전복귀 | 관제 판단은 이 정도면 충분 |
| **0.5s** | `ui.timer` → `dash.refresh()` | 화면 그리기 | 20Hz 로 그리면 브라우저가 죽는다 |

**이걸 하나로 합치면 안 된다.** 제어를 0.5s 로 하면 로봇이 끊기고, 화면을 20Hz 로 그리면 브라우저가 죽는다.

그 대가로 **스레드가 2개**가 되고, 둘이 같은 `RobotState` 를 만진다 → `FLEET_LOCK`(RLock).
여기서 **3가지 규칙**이 나왔다. 셋 다 **실제로 물려서** 알게 된 것이다:

1. **A\* 는 락 밖에서 계산한다.**
   락을 쥔 채 경로계획(수십 ms)을 돌리면 20Hz 제어루프가 그동안 멈춰 로봇이 끊긴다.
   → 계산은 밖, **대입만** 안에서.
2. **화면은 `state.snapshot()` 사본으로 그린다.**
   살아있는 `r.path` 를 읽으면, `if r.path` 를 통과한 직후 제어루프가 `pop(0)` 해서
   `r.path[-1]` 이 `IndexError` 로 터진다. **로봇이 목표에 도착할 때마다 열리는 창**이라 반드시 터진다.
3. **관제틱은 `app.timer`(앱 1개), 화면갱신은 `ui.timer`(페이지별).**
   → §4 참고.

## 4. 실제로 물린 버그들 (이 노트의 핵심)

> 전부 **동작하는 것처럼 보였다.** 그래서 위험했다.

### 4-1. 거북이가 엉뚱한 자리에 spawn — *콜백이 입력값을 덮어썼다*
`turtle1` 만 config 의 `(2,2)` 가 아니라 `(5.54, 5.54)` 에 생겼다. turtle2~4 는 멀쩡.

**원인**: `/turtle1/pose` **구독을 kill 보다 먼저** 만들었다. kill 서비스 응답을 기다리며
`spin_until_future_complete` 가 도는 동안 **기존 turtle1 의 pose 콜백이 `r.x` 를 5.54 로 덮어썼고**,
그 값이 그대로 spawn 좌표로 들어갔다. turtle2~4 는 아직 존재하지 않아 콜백이 없었다 → 멀쩡.

**교훈**: 서비스 호출 중에 spin 하면 **콜백이 돈다.** 그때 내 입력 데이터가 바뀔 수 있다.
**고침**: ① 토픽 개설을 spawn **이후**로 옮김 ② spawn 좌표를 미리 스냅샷.

### 4-2. 종료 시 core dump — *`terminate called without an active exception`*
`_exec.shutdown()` 뒤 스레드를 **join 하지 않고** `destroy_node()` 를 했다.
spin 중인 스레드를 두고 노드를 파괴하면 C++ 쪽에서 terminate.
**고침**: shutdown → **join(timeout=3)** → destroy → rclpy.shutdown. 순서가 전부다.

### 4-3. E-STOP 이 E-STOP 이 아니었다 — *가장 위험했던 버그*
전 로봇을 `idle` 로 떨어뜨리고 작업을 취소했다. 그런데 **0.5초 뒤 로봇들이 스스로 다시 출발**했다.
`_auto_charge()` 가 "idle + 저전력" 로봇을 충전소로 보내고, `dispatch()` 가 남은 작업을 배차해서.

**교훈**: **정지는 "상태를 바꾸는 것"이 아니라 "래치를 거는 것"이다.**
자동 로직이 살아있는 한, 상태만 바꿔놓으면 그 로직이 원상복구시킨다.
**고침**: `estopped` 래치 + `[🔓 해제(RESET)]`. 래치 중엔 자동(배차·충전)·수동(이동·teleop) 전부 거부.
(단 **■ 개별정지는 허용** — 정지는 언제나 안전한 명령.)

### 4-4. 브라우저를 닫으면 관제가 멈췄다 — *NiceGUI 의 함정*
```python
@ui.page("/")
def index():
    ui.timer(state.poll_interval, lambda: (state.refresh(), dash.refresh()))  # ❌
```
`@ui.page` 함수는 **브라우저마다** 실행된다. 그래서 `state.refresh()`(= 관제 두뇌)가:
- 브라우저 **0개** → **아예 안 돎.** 작업을 넣어도 배차가 안 된다.
- 브라우저 **2개** → 배차·배터리가 **2배속**.

**교훈**: 화면 프레임워크의 타이머에 **도메인 로직**을 얹지 마라. 화면은 관전자여야 한다.
**고침**: `app.timer(…, state.refresh)`(앱 1개) + `ui.timer(…, dash.refresh)`(페이지별).
**검증**: 브라우저 0개로 2초 → 관제틱 10회 확인.

### 4-5. 정지 버튼 두 번으로 작업이 "완료"됐다 — *`idle` 을 도착으로 믿었다*
```python
if r.state == "idle":   # ❌ "도착했구나"
    t.phase = "to_drop"
```
`idle` 이 되는 경로는 도착 말고도 있다: **정지(■)·teleop 만료·통신두절**.
→ 주행 중 ■ 를 누르면 "픽업 완료 → 드롭으로 출발", 한 번 더 누르면 **"작업 완료"**.
거북이는 **어디에도 간 적이 없는데** 큐에는 성공으로 남는다.

**교훈**: **상태(state)로 사건(event)을 추론하지 마라.** 여러 원인이 같은 상태로 수렴한다.
**고침**: "목표 스테이션 반경 `ARRIVE_EPS`(0.4) 안에 실제로 서 있나"를 **좌표로 확인**.
아니면 `_requeue()` 로 작업을 큐에 돌려보낸다(다른 로봇이 이어받음).

### 4-6. 죽은 로봇의 작업이 영원히 `running` 이었다
`online=False` 는 표시했지만 `state` 는 `driving` 그대로, `path` 도 그대로.
→ 맵에 **죽은 로봇의 유령 경로**가 계속 그려지고, 그 작업은 아무도 안 건드린다.
**고침**: `was_online and not online` → `error` + `path=[]`.
(`was_online` 검사가 중요: 이게 없으면 **기동 직후 첫 pose 를 기다리는 동안 전 로봇이 error** 가 된다.)
그리고 `_advance_tasks()` 가 그 작업을 `pending` 으로 되돌린다 → 다음 `dispatch()` 가 재배차.
**복구는 수동**(■ 을 눌러 idle 로) — 되살아난 로봇이 옛 명령으로 갑자기 튀어나가지 않게.

### 4-7. A* 가 벽을 뚫는 경로를 정상인 척 내줬다
경로가 없으면 `return [goal]` (직선 폴백) 이었다. **"최소 동작 보장"** 이라고 생각했는데,
호출자는 그게 폴백인지 알 수가 없다 → 벽을 통과하는 경로를 맵에 점선으로 당당히 그린다.
**교훈**: **못 하면 못 한다고 말해야 한다.** 조용한 폴백은 거짓말이다.
**고침**: `return []` → `send_goal()` 이 목표를 거부 → UI 가 "경로가 없습니다 (벽에 막힘)".

### 4-8. 웹이 안 켜졌다 — *포트를 물고 있던 좀비*
테스트로 띄운 서버가 안 죽고 8090 을 점유. 새로 켜면 `address already in use` 로 죽는데,
브라우저는 **낡은 서버**가 응답하니 "켜지긴 하는데 안 바뀐다"로 보인다. 최악의 증상.
**고침**: `run.sh` 가 포트 점유를 감지해 **PID 와 `kill` 명령까지** 알려준다.

### 4-9. `run.sh` 가 깨진 venv 를 영원히 안 고쳤다
설치를 `if [ ! -d .venv ]` 로 감쌌다. venv 만 만들어지고 pip 이 실패하면(네트워크 등),
다음부터 "폴더 있네" 하고 설치를 건너뛰어 `ModuleNotFoundError` 로만 죽는다 — 원인도 안 보인 채.
**교훈**: **존재 여부로 완료 여부를 판단하지 마라.**
**고침**: venv 생성만 조건부, `pip install` 은 **매번**(이미 깔렸으면 즉시 끝난다).

## 5. 환경 함정: python 3.13 vs 3.12

시스템 기본 python 은 **3.13**, Jazzy 의 `rclpy` 는 **3.12 전용 빌드**.
3.13 으로 import 하면 `_rclpy_pybind11.cpython-313-….so isn't present`.
→ `python3.12 -m venv --system-site-packages` + ROS `setup.bash` source.
`run.sh` 가 전부 자동으로 한다. (상세: `docs/환경.md` §1)

## 6. 교통관제를 이렇게 설계한 이유

**요구**: 거북이끼리 부딪히지 않게. 실제 AMR 관제의 최난제.

**흔한 설계**(구역 예약)는 zone 분할 + mutex + **교착 감지·해소**가 필요하다.
교착 해소는 그 자체로 어렵다(대기 그래프 순환 탐지 → 후퇴 → 재계획).

**그래서 교착이 생길 수 없는 규칙을 골랐다**:
> 우선순위를 **전순서(total order)** 로 준다(config 순서). 두 로봇이 가까워지면 **낮은 쪽이 선다.**

"A는 B를, B는 A를 기다리는" 상황이 **구조적으로 불가능**하다 → 교착 감지 코드가 **아예 필요 없다**.
여기에 하나 더: **다가가는 중일 때만** 세운다(내적 > 0). 안 그러면 뒤따라가는 로봇이 영영 못 간다.

### 한계 — 정직하게
**정면 대향은 못 푼다.** 실측 최소 접근거리 **0.15**(safe_dist=1.2 인데).
양보하는 쪽은 서지만 **선 자리가 곧 상대의 경로**라 우선순위 로봇이 그대로 통과한다.
(turtlesim 은 거북이끼리 물리 충돌이 없어 겹쳐도 아무 일도 안 일어난다.)

→ 지금 수준은 **"충돌을 막는다"가 아니라 "관제가 교통을 정리한다"**.
제대로 풀려면 선 로봇을 임시 장애물로 넣고 **A\* 재계획(우회)** 하거나 일방통행 규칙이 필요.
(`구현_프롬프트.md` [P4-2], 완료기준: min_dist > safe_dist)

**서 있는(idle) 로봇은 장애물로 안 친다.** 치면 그 앞 로봇이 영원히 못 간다(비켜줄 수가 없으므로).

## 7. 화면 설계에서 배운 것

### 폼은 자동갱신 밖에 둬야 한다
대시보드는 0.5초마다 통째로 다시 그린다. `ui.select` 를 그 안에 두면 **매 0.5초 선택이 리셋**된다.
→ 조작 패널·작업생성 폼은 **한 번만** 렌더(refresh 밖), 읽기전용만 refresh 안.
그래서 조건부 표시(해제 버튼)는 `if` 가 아니라 **바인딩**으로 해야 한다:
`ui.button(…).bind_visibility_from(state, "estopped")`.

### 색이 두 가지를 뜻하면 안 된다
처음에 로봇 펜 색 팔레트가 `#2e86de` 로 시작했는데, 이게 **`STATE_COLORS["driving"]` 과 완전히 같았다.**
화면의 파란 선이 "1번 로봇"인지 "주행중"인지 알 수가 없다.
→ 펜 색은 **상태색과 절대 안 겹치게** + 파란 배경에서 잘 보이게(노랑/분홍/청록/주황) 다시 골랐다.
**단일 정의처**(`models.py`)에서 맵뷰·카드가 같이 import 한다. UI 파일마다 하드코딩하면 화면끼리 어긋난다.

### 관제는 "왜 그랬는지"를 보여줘야 한다
로봇이 그냥 멈춰 있으면 운영자는 **고장으로 오해**한다.
→ 양보 중이면 주황 링 + 안전거리 원 + 위험 쌍을 빨간 점선으로 잇는다.
같은 이유로, 명령이 거부되면 **조용히 무시하지 않고** 이유를 알린다("E-STOP 상태입니다", "경로가 없습니다").

## 8. 검증 방식

**"코드 짰음"은 완료가 아니다. 실제 거북이가 움직인 증거만 인정.**

| 스위트 | 무엇 |
|---|---|
| 기본 e2e | 목적지→A*→실제 도착 / teleop→이동 후 자동정지 / 자동배차 / E-STOP |
| 버그수정 e2e | 정지를 도착으로 안 믿나 / E-STOP 래치 / snapshot 독립성 / 도달불가 거부 |
| 교통관제 e2e | 정면대향 시 양보 발생 / 우선순위 규칙 / 교착 없음 / 재개·도착 / 불필요한 정지 없음 |

**테스트 인프라에서 물린 것**: `pkill -f 'turtlesim_node'` 가 **자기 자신의 명령줄**을 매칭해서
스크립트가 자살했다(exit 144). 명령줄에 그 문자열이 들어 있으면 `pgrep/pkill -f` 는 자기도 잡는다.
→ 프로세스 정리는 **별도 호출**로 분리. (`pgrep` 오탐으로 "turtlesim 실행중"이라 착각한 적도 있다.)
