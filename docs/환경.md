# 런타임 / 개발 환경 정의 (FMS TurtleBot Fleet)

> **원칙: OS에 맞는 ROS 2 distro를 네이티브 설치.** Ubuntu **22.04 → Humble**, **24.04 → Jazzy**. distro가 OS와 맞으면 **Docker 불필요.** 팀원 머신 OS가 달라도 같은 소스를 각자 자기 distro로 빌드한다.
> (README의 "Humble 타깃 / Docker Humble" 항은 이 결정으로 갱신됨 — Docker는 OS가 맞지 않을 때의 대안일 뿐.)
> 미확정 조합은 "확인 필요"로 표기(창작 금지).

---

## 0. 이 머신 실측 (2026-07, 확인됨)
| 항목 | 값 | 근거 |
|---|---|---|
| OS | **Ubuntu 24.04 Noble** | `lsb_release` |
| 설치된 ROS | **Jazzy** (`/opt/ros/jazzy`, `ROS_DISTRO=jazzy`) | 실측 |
| Docker | 미설치 — **Jazzy 네이티브라 불필요** | `docker --version` 실패 |
| GPU / DISPLAY | RTX 5090 / `DISPLAY=:1` | `nvidia-smi` |
| 필요 스택 apt 가용성 | **전부 Jazzy용 존재**(§2) | `apt-cache policy` 실측 |

→ **이 머신 = 24.04 → Jazzy 네이티브로 진행. Docker 없음.**

## 1. OS별 distro 규칙 (핵심)
| OS | ROS 2 distro | Gazebo | 설치 |
|---|---|---|---|
| Ubuntu 22.04 Jammy | **Humble** | Gazebo Classic 11 (`gazebo_ros`) 또는 gz | apt `ros-humble-*` 네이티브 |
| Ubuntu 24.04 Noble | **Jazzy** | **gz sim (Harmonic) + `ros_gz`** | apt `ros-jazzy-*` 네이티브 |

- 각 머신은 **자기 OS에 맞는 distro**를 깐다(호스트에 안 맞는 distro 네이티브 설치 금지, 크로스 금지).
- 프로젝트 코드(launch/config/노드)는 **distro 무관하게 작성** — 패키지명 prefix(`ros-humble-`/`ros-jazzy-`)만 OS별로 다름.
- OS를 못 맞추는 경우에만(예: 22.04 타깃인데 24.04 머신) **Docker**로 해당 distro 컨테이너 사용(대안).

## 2. 필요 패키지 (이 머신 = Jazzy 기준, apt 가용성 실측 ✓)
| 패키지 (Jazzy) | 용도 | 22.04면 |
|---|---|---|
| `ros-jazzy-turtlebot3`(-msgs) | 로봇모델·메시지 | `ros-humble-turtlebot3` |
| `ros-jazzy-turtlebot3-simulations` / `-gazebo` | 시뮬·world·bringup | `ros-humble-...` |
| `ros-jazzy-navigation2` / `-nav2-bringup` | 주행·경로계획·회피 | `ros-humble-...` |
| `ros-jazzy-slam-toolbox` | SLAM 맵작성 | 동일 |
| `ros-jazzy-ros-gz` | Gazebo(gz sim) 연동 | Humble은 `gazebo_ros`(Classic) 확인 |
| `ros-jazzy-rosbridge-server` | ROS↔웹 브릿지(옵션) | 동일 |
| env `TURTLEBOT3_MODEL` | burger / waffle / **waffle_pi**(lidar+cam 권장) | — |

**이 머신 설치 명령(sudo)**:
```
sudo apt update && sudo apt install -y \
  ros-jazzy-turtlebot3 ros-jazzy-turtlebot3-simulations ros-jazzy-turtlebot3-gazebo \
  ros-jazzy-navigation2 ros-jazzy-nav2-bringup ros-jazzy-slam-toolbox ros-jazzy-ros-gz
```
- Gazebo: Jazzy는 **gz sim(Harmonic) + `ros_gz`** 조합(`ros-jazzy-ros-gz` 존재 확인). Gazebo Classic 아님.

## 3. 다중 로봇 (네임스페이스) — distro 무관
- 로봇별 네임스페이스 `/tb1 /tb2 …`, tf 프레임 prefix, topic/액션 remap (README §2).
- **DDS**: 같은 fleet = 같은 `ROS_DOMAIN_ID` + 네임스페이스 분리(일반 권장 — 이 프로젝트 정책은 P2에서 확정, 확인 필요).
- **RMW**: 기본 `rmw_fastrtps`, 대수 늘면 `rmw_cyclonedds` 검토(확인 필요).

## 4. 개발 vs 배포
| 구분 | 개발(시뮬) | 배포(실기) |
|---|---|---|
| 로봇 | Gazebo TurtleBot3 N대 (이 워크스테이션 **Jazzy 네이티브**) | 실 TurtleBot3 N대 |
| 센서 | 시뮬 LiDAR/odom | 실 LiDAR/IMU(노이즈·지연, README §7-9) |
| 표시 | 호스트 DISPLAY(RTX5090) 네이티브 렌더 | 관제 PC 웹 브라우저 |
| 리스크 | 낮음 | 로컬 드리프트·통신·안전(실기 게이트) |

- **시뮬↔실기 갭(README §7-9)**: 센서 노이즈·지연·마찰 → 파라미터 재튜닝(확인 필요). P1~P5는 우선 시뮬 검증(README §9).
- **실기 전환 시**: `fable-work-doctrine` §5(실기 하드웨어 게이트) — 라이브 증거 없이 완료 선언 금지.

## 5. GPU / DISPLAY (실측)
- RTX5090 + `DISPLAY=:1` → **gz sim / RViz2를 네이티브로 직접 렌더**(Docker GPU 패스스루 불필요 — 네이티브라 그대로 사용).
