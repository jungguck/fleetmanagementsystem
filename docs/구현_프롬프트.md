# FMS 구현 태스크 / 프롬프트 라이브러리

> 마스터, 이 문서는 `README.md`의 로드맵 **P0~P5를 자기완결적 소태스크**로 쪼갠 것입니다.
> 각 태스크는 그대로 **구현 프롬프트**로 붙여 쓸 수 있도록 `[Pn-k] 제목 / 목적 / 할 일 / 재사용(경로) / 산출물 / 완료기준(검증)` 형식입니다.
> 환경 버전 미확정 사항은 `환경.md`를 정본으로 참조하고, 여기서는 "확인 필요"로 둡니다. README와 모순되는 결정은 없습니다.

**공통 규약**
- 타깃 ROS 2 **Humble**, 개발머신 Jazzy/24.04 → **Docker Humble**(`환경.md`).
- 로봇 네임스페이스 `/tb1 /tb2 …`. 공유 맵 `map` 프레임.
- 관제 레이어는 재사용(`~/plc_study_for_me`, `~/robot_api`), **TurtleBot↔관제 브릿지만 신규**(README §6).
- 각 태스크 완료기준은 **라이브 실행 증거**(로그/명령/도착)로만 통과.

## P0 — 환경 (완료기준: 시뮬 1대 뜸)

### [P0-1] ROS 2 스택 설치 (OS에 맞는 distro 네이티브 — Docker 불필요)
- **목적**: 시뮬 스택 확보. **OS별로 22.04→Humble / 24.04→Jazzy 네이티브**(이 머신=24.04→Jazzy).
- **할 일** (sudo, 마스터 실행): `sudo apt install ros-<distro>-{turtlebot3, turtlebot3-simulations, turtlebot3-gazebo, navigation2, nav2-bringup, slam-toolbox, ros-gz}`. `TURTLEBOT3_MODEL=waffle_pi` env. (OS가 distro와 안 맞을 때만 Docker로 해당 distro 컨테이너 — 대안.)
- **재사용**: 없음. 참조 `환경.md` §1~2.
- **산출물**: `setup/install.sh`(distro 자동감지), `env.sh`(모델/도메인ID).
- **완료기준**: `ros2 pkg list | grep -E "turtlebot3|nav2"` 존재, 의존성 해소.

### [P0-2] FMS 워크스페이스 스캐폴드 + env
- **목적**: 프로젝트 코드 골격(distro 무관 작성).
- **할 일**: `fms_ws/src/` colcon 패키지 골격(bringup·fleet·gui), `env.sh`(`TURTLEBOT3_MODEL`, `ROS_DOMAIN_ID`), 실행법 README.
- **재사용**: 없음.
- **산출물**: `fms_ws/`(빈 패키지라도), `env.sh`.
- **완료기준**: `colcon build` 성공.

### [P0-3] Gazebo world + TurtleBot3 sim bringup (1대)
- **목적**: 시뮬 1대 구동.
- **할 일**: TurtleBot3 기본 world 또는 창고형 world 준비, bringup 런치. odom/scan/tf 확인.
- **재사용**: 없음.
- **산출물**: `launch/sim_bringup.launch.py`, `worlds/warehouse.world`(또는 기본).
- **완료기준**: Gazebo에 로봇 1대 스폰, `ros2 topic echo /scan` 데이터, `ros2 topic echo /odom` pose 갱신.

### [P0-4] nav2 단독 기동 확인
- **목적**: nav2 스택 정상 동작 확인.
- **할 일**: `nav2_bringup` + 기본 맵으로 nav2 실행, RViz에서 2D Goal Pose 전송.
- **재사용**: 없음.
- **산출물**: `launch/nav2.launch.py`, params yaml(기본값 시작).
- **완료기준**: RViz 목표 클릭 → 로봇이 그 지점 도착(P1 예행).

## P1 — 단일 (완료기준: 클릭→로봇 그 지점 도착)

### [P1-1] SLAM 맵 작성 + 저장
- **목적**: 공유 맵 확보(README §7-2).
- **할 일**: `slam_toolbox`로 world 매핑, teleop 주행으로 커버, `map_saver_cli`로 pgm/yaml 저장.
- **재사용**: teleop 참조 `~/mobile_robot_proto_type-main/.../hardware_test.py`(cmd_vel Twist 발행 패턴).
- **산출물**: `maps/warehouse.yaml`, `warehouse.pgm`.
- **완료기준**: 저장 맵으로 AMCL 로컬라이즈 성공(RViz에서 pose 수렴).

### [P1-2] AMCL 로컬라이제이션 브링업
- **목적**: 저장 맵 위 위치추정.
- **할 일**: nav2 localization(amcl) 런치, 초기포즈 설정.
- **재사용**: nav2_bringup.
- **산출물**: `launch/localization.launch.py`.
- **완료기준**: 주행 중 pose 드리프트 없이 맵 정합.

### [P1-3] nav2 목표전송 노드 (rclpy)
- **목적**: 프로그램적으로 목표 전송(NavigateToPose 액션).
- **할 일**: rclpy 노드/모듈로 `NavigateToPose` 액션 클라이언트 작성, `(x,y,yaw)` 입력 → goal 전송 → 결과/피드백 수신. 네임스페이스 파라미터화.
- **재사용**: 없음(브릿지 신규). 구조는 hardware_test의 rclpy 노드 패턴 참고.
- **산출물**: `fms_bridge/nav_client.py`.
- **완료기준**: CLI로 좌표 주면 로봇 도착, 완료 콜백 로그.

### [P1-4] 웹 백엔드 ↔ ROS 브릿지 (rclpy)
- **목적**: 관제(웹)에서 ROS로 명령/상태 왕복(README §7-4).
- **할 일**: 백엔드 프로세스에 rclpy 노드 임베드(별도 executor 스레드). 명령 API(goal 전송) + 상태 구독(pose/odom → 캐시). rosbridge vs rclpy직접은 `환경.md` 결정포인트(직접 채택 권장, 확인 필요).
- **재사용**: `~/plc_study_for_me/app/state.py`(refresh 루프·snapshot·`_fetch_bounded` 폴링 구조), `~/robot_api/api/server.py`(API 서버 골격).
- **산출물**: `fms_backend/ros_bridge.py`, `fms_backend/state.py`(LineSnapshot 대응 FleetSnapshot).
- **완료기준**: HTTP/함수 호출 → 로봇 이동, 상태 폴링에 라이브 pose 반영.

### [P1-5] GUI 목표전송 버튼 + 1대 상태카드
- **목적**: GUI에서 1대 관제(README §P1).
- **할 일**: NiceGUI 화면에 맵뷰(간이) + 목표전송(맵 클릭 또는 좌표 입력) + 상태카드 1장. `dashboard_body` @ui.refreshable 갱신.
- **재사용**: `~/plc_study_for_me/app/ui/dashboard.py`(레이아웃·`_panel`·refreshable), `ui/components.py`(`robot_card` → 로봇 상태카드로 치환). 상세 매핑은 `GUI_설계.md`.
- **산출물**: `fms_backend/ui/dashboard.py`, `ui/components.py`(robot_card 개조).
- **완료기준**: **GUI 클릭 → Gazebo TurtleBot이 목표점 도착**(README §9 e2e 게이트).

### [P2-1] 네임스페이스 다중 로봇 sim bringup

## P3 — 작업/배차 (완료기준: 작업 넣으면 로봇 자동 선택·수행)

### [P3-1] 스테이션·맵 모델
- **목적**: PICKUP/DROP/CHARGE/IDLE 좌표 정의(README §3).
- **할 일**: 스테이션 YAML(id, type, x, y, yaw). 로더.
- **재사용**: `~/plc_study_for_me/config.yaml`·`app/config.py`(YAML 설정화 패턴).
- **산출물**: `config/stations.yaml`, `config/robots.yaml`, `fms_backend/config.py`.
- **완료기준**: 로드된 스테이션이 맵뷰에 마커로 표시.

### [P3-2] 작업큐 + 작업 상태머신
- **목적**: 작업 생성·추적(시나리오 Task FSM).
- **할 일**: Task 모델(id, pickup, drop, priority, state) + FSM(CREATED→…→DONE). 큐 자료구조.
- **재사용**: 없음(신규 로직).
- **산출물**: `fms_backend/task_queue.py`, `fms_backend/task_fsm.py`.
- **완료기준**: 작업 생성/전이/완료가 로그·상태로 추적됨.

### [P3-3] 배차 로직 (근접/한가)
- **목적**: 자동 배차(README §3, §7-3).
- **할 일**: 가용 로봇 중 score(거리·배터리·혼잡) 최소 선택. 단순 근접부터 시작. 점유 락(동일 PICKUP).
- **재사용**: 없음.
- **산출물**: `fms_backend/dispatcher.py`.
- **완료기준**: 작업 투입 → 가장 가깝고 한가한 로봇 자동 배정·수행(시나리오2).

### [P3-4] 작업 실행 오케스트레이터
- **목적**: FSM ↔ nav2 연결(GOTO/LOADING/UNLOADING).
- **할 일**: FSM 단계별 nav_client goal 발행, 적재/하역 상태 플래그 처리(포크 없음, README §1). 재배차 훅(시나리오6·7).
- **재사용**: `state.py`의 제어 루프(refresh) 구조.
- **산출물**: `fms_backend/orchestrator.py`.
- **완료기준**: 작업 1건이 A→적재→B→하역→복귀 자동 완주.

### [P3-5] GUI 작업 생성/관리 패널
- **목적**: 작업 CRUD·우선순위(시나리오6).
- **할 일**: 작업생성 폼(PICKUP/DROP/priority), 큐/진행/완료 리스트, 우선순위 변경.
- **재사용**: `ui/components.py`(리스트·배지 패턴), `error_list` 스타일.
- **산출물**: GUI 작업패널. 상세는 `GUI_설계.md`.
- **완료기준**: GUI로 작업 넣으면 로봇 자동 선택·수행 관측.

## P4 — 교통관제 (완료기준: 교차로에서 안 부딪힘, 최난제 README §7-1)

### [P4-0] 결정 게이트: Open-RMF vs 커스텀 예약
- **목적**: README §5/§8 핵심 결정 확정.
- **할 일**: Open-RMF TurtleBot 데모 PoC vs 커스텀 구역예약 난이도·안정성 비교, 채택 결정.
- **재사용**: Open-RMF(외부).
- **산출물**: 결정 기록(ADR) 1장.
- **완료기준**: 채택안 확정(확인 필요). 아래 태스크는 커스텀 경로 기준, RMF 채택 시 대체.

### [P4-1] 구역(zone) 모델 + 예약 매니저

- **할 일**: 맵을 교차로/좁은통로 zone으로 분할, zone별 예약(1대) 관리자.
- **재사용**: 없음.
- **산출물**: `config/zones.yaml`, `fms_backend/traffic/reservation.py`.
- **완료기준**: 동일 zone 동시 요청 시 1대만 승인, 나머지 대기.

### [P4-2] 진입 게이팅 + 감속/정지 연동
- **목적**: 예약 미승인 로봇 zone 앞 정지.
- **할 일**: nav2 goal/speed 제어로 zone 경계 정지, 승인 시 재개(PAUSED 상태).
- **재사용**: nav_client, orchestrator.
- **산출물**: `fms_backend/traffic/gating.py`.
- **완료기준**: 교차로에서 로봇들이 순차 통과, 충돌 없음.

### [P4-3] 데드락 감지·해소
- **목적**: 상호대기 해소(시나리오3).
- **할 일**: 예약 대기 그래프 순환 감지 → 저순위 후퇴/재계획.
- **재사용**: 없음.
- **산출물**: `fms_backend/traffic/deadlock.py`.
- **완료기준**: 대향 통로 교착 상황이 자동 해소.

### [P4-4] 로봇 pose 동적 장애물 공유
- **목적**: 상대 로봇 회피 개선(시나리오3 예외).
- **할 일**: 각 로봇 pose를 costmap obstacle layer로 공유(확인 필요).
- **재사용**: nav2 costmap 플러그인.
- **산출물**: costmap params 확장.
- **완료기준**: 근접 시 지역 회피 개선 관측.

## P5 — 적재/충전 (완료기준: 왕복 사이클 무인 반복)

### [P5-0] 적재 검증 방식 결정 게이트
- **목적**: 포크 없는 적재/하역 표현 확정(README §7-5).
- **할 일**: ①상태 플래그 ②AR마커 정렬 ③간이 메커니즘 중 채택.
- **산출물**: 결정 기록(ADR).
- **완료기준**: 채택안 확정(확인 필요).

### [P5-1] 적재/하역 상태머신 구현
- **목적**: LOADING/UNLOADING 상태 처리(시나리오1).
- **할 일**: PICKUP 도착 정렬 → 적재 플래그 set(+검증), DROP 도착 → clear. 실패 재시도.
- **재사용**: orchestrator FSM.
- **산출물**: `fms_backend/loading.py`.
- **완료기준**: 적재 상태가 GUI·로그에 반영, 검증 실패 시 재시도/에러.

### [P5-2] 자동 충전복귀
- **목적**: 배터리 관리(시나리오4, README §7-6).
- **할 일**: battery 임계(LOW/CRIT, 확인 필요) 감지 → CHARGE 스테이션 예약·이동·충전·복귀. 간이 도킹.
- **재사용**: dispatcher(충전소 배정), orchestrator.
- **산출물**: `fms_backend/charging.py`.
- **완료기준**: 저배터리 로봇이 자동 충전 후 작업 복귀.

### [P5-3] 안전: E-STOP + 사람/장애물 대응
- **목적**: 안전(시나리오5·8, README §7-7).
- **할 일**: fleet broadcast E-STOP(cmd_vel=0, nav 취소, ESTOP 상태), 부분실패 집계, 2단계 RESET. 장애물 debounce.
- **재사용**: `~/robot_api/api/server.py` `/api/fleet/emergency_stop` + `robot_manager.py`(units 병렬, partial-failure 반환 dict). 참고: `~/robot_api/ESTOP_SAFETY_HANDOFF_KR.md`.
- **산출물**: `fms_backend/safety.py`, GUI E-STOP 버튼.
- **완료기준**: E-STOP 클릭 → 전 로봇 즉시 정지, 미응답 로봇 개별 알람, RESET로만 복귀.

### [P5-4] 무인 왕복 사이클 회귀 테스트
- **목적**: README §P5 최종 완료기준.
- **할 일**: N대·다작업·충전·교통 포함 시나리오를 무인 반복 실행.
- **재사용**: 전체.
